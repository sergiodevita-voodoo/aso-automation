# VoodooStudios/aso-automation

Shared GitHub Action that drives the biweekly ASO automation (runs 1st + 15th of each month) for every PGT game:

- Fresh game icon generated by the team's Scenario AI workflow
  (`Icon Workflow fo AI ASO`), reading every per-node parameter live
- Fresh App Store / Google Play "What's New" text generated via Claude
- Unity icon variants overwritten in place (`.meta` GUIDs preserved)
- Release branch cut from `develop`, CircleCI iOS + Android builds triggered
- Apple build attached to a new ASC version, emojis stripped, submitted via
  `reviewSubmissions`, `releaseType=AFTER_APPROVAL` (auto-publishes on review)
- Google Play promoted Internal → Production at 100% rollout, with the new
  store-listing icon uploaded as a separate asset (Play has two icon stores)
- Release branch merged back to `develop` + `main` per PGT branching convention
- **Auto-rollback** on failure — every side-effect tracked, reversed best-effort

All automations across all games execute this exact same code. Per-game
divergence is config-only (in each game's `aso-automation.config.yml`).

---

## How a game uses this

Each PGT game's repo has two files at the root:

1. **`aso-automation.config.yml`** — game-specific values (game name, ASC app id,
   Play package, icon paths, CircleCI deployer branch, Slack channel)
2. **`.github/workflows/aso-biweekly.yml`** — a 30-line wrapper that calls this
   action on the biweekly cron (`0 9 1,15 * *` UTC) + via `workflow_dispatch`

Templates for both live in [`./templates/`](./templates/). The
`pgt-setup-aso-automation` skill in the
[claude-skills-marketplace](https://github.com/VoodooStudios/claude-skills-marketplace)
renders them per game and commits to the game's `develop` branch.

## Secrets (org-level, never in repos)

Six secrets are scoped to PGT game repos at the `VoodooStudios` org level. Game
repos never store or display the values — they inherit them.

| Secret | Where it comes from | Per-game? |
|---|---|---|
| `SCENARIO_API_KEY` | Voodoo Scenario API key | shared |
| `CIRCLECI_TOKEN` | CircleCI PAT (`CCIPAT_*`) | shared |
| `ASC_KEY_ID` | App Store Connect API key ID | shared |
| `ASC_PRIVATE_KEY_P8` | App Store Connect `.p8` contents | shared |
| `GOOGLE_PLAY_SA_JSON` | `icon-updater@pgt-icon-update.iam.gserviceaccount.com` SA JSON | shared (must be granted on each game's Play Console) |
| `ANTHROPIC_API_KEY` | Voodoo Anthropic API key | shared |

## Versioning

The action is pinned by tag in each game's workflow:

```yaml
- uses: VoodooStudios/aso-automation@v1
```

Push hotfixes to `v1` to update all games at once. Breaking changes go to `v2`.

## Hard-won lessons baked in (don't re-discover these)

1. **Apple's "What's New" REJECTS emojis.** Stripped automatically before ASC; Play accepts them — kept.
2. **Use `reviewSubmissions`, NOT `appStoreVersionSubmissions`.** The latter is deprecated and returns confusing 403s.
3. **`releaseType: AFTER_APPROVAL` on ASC `create_version`** — auto-publishes on Apple Review approval. Without it, the version sits forever waiting for a human click.
4. **The Play Store *listing* icon is a separate asset** from the AAB's launcher icon. Must be uploaded via `edits.images().upload(imageType="icon")`. Otherwise the Play Store page shows the OLD icon even after a successful release.
5. **Strip Claude meta-annotations** like `*(chars: 72 — well within 500)*` from the generated What's New — Claude likes to append self-commentary.
6. **Don't mention the icon in the What's New text.** Players don't see the icon update as news; the marketing copy should focus on gameplay feel.
7. **Cut release from `develop`, merge back to both `develop` + `main`/`master`** per PGT branching convention.
8. **iOS build numbers can't be re-used** even after expiry — orchestrator always bumps to N+1 where N is the highest existing.
9. **CC deployer config branch differs per game** (`feature/<GameName>_<UnityVersion>`) — must exist before the ASO automation can trigger builds.
10. **Auto-rollback on any failure.** Every side-effect (branch pushed, ASC version, attached build, Play Internal release) is tracked in `_RunState` and reversed best-effort if any step fails.
